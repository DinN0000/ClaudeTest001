# PRD 작성 체크리스트

PRD를 작성하거나 검토할 때 다음 체크리스트를 활용하세요.

---

## 📝 작성 전 (Pre-Writing)

### 준비 단계
- [ ] 프로젝트의 목적과 배경을 명확히 이해했는가?
- [ ] 주요 이해관계자를 파악하고 연락했는가?
- [ ] 유사한 프로젝트나 경쟁사 사례를 조사했는가?
- [ ] 사용자 리서치나 데이터 분석을 완료했는가?
- [ ] 템플릿을 적절한 위치에 복사했는가? (feature/bugfix/improvement)

### 파일 설정
- [ ] 파일명이 네이밍 컨벤션을 따르는가? (`YYYY-MM-DD_프로젝트명_PRD.md`)
- [ ] 문서 상단의 메타데이터(작성자, 날짜, 버전)를 작성했는가?
- [ ] 초기 상태를 "🟡 Draft"로 설정했는가?

---

## 📄 내용 작성 (Content)

### 1. 개요 섹션
- [ ] 프로젝트를 2-3문장으로 명확히 요약했는가?
- [ ] 해결하려는 문제가 구체적으로 정의되었는가?
- [ ] 해결 방안이 명확히 제시되었는가?

### 2. 배경 및 목표
- [ ] 프로젝트가 필요한 비즈니스적 배경을 설명했는가?
- [ ] 측정 가능한 목표를 설정했는가?
- [ ] In Scope와 Out of Scope가 명확히 구분되었는가?
- [ ] 범위 설정에 대해 이해관계자와 합의했는가?

### 3. 이해관계자
- [ ] 모든 주요 이해관계자를 나열했는가?
- [ ] 각 이해관계자의 역할과 책임이 명확한가?
- [ ] 연락처 정보가 정확한가?

### 4. 사용자 스토리
- [ ] 주요 사용자 페르소나를 정의했는가?
- [ ] 각 스토리가 "누가, 무엇을, 왜" 형식을 따르는가?
- [ ] 모든 스토리에 Acceptance Criteria가 있는가?
- [ ] 우선순위가 명확히 표시되었는가?
- [ ] 사용자 스토리 간 의존성을 파악했는가?

### 5. 기능 요구사항
- [ ] 모든 핵심 기능에 고유 ID(REQ-XXX)를 부여했는가?
- [ ] 각 요구사항이 구체적이고 측정 가능한가?
- [ ] 우선순위(P0/P1/P2)가 명확한가?
- [ ] Acceptance Criteria가 명확하게 정의되었는가?
- [ ] 엣지 케이스를 고려했는가?

### 6. 비기능 요구사항
- [ ] 성능 요구사항(응답 시간, 처리량)이 구체적인가?
- [ ] 확장성 요구사항이 정의되었는가?
- [ ] 가용성 목표(SLA)가 설정되었는가?
- [ ] 보안 요구사항이 명시되었는가?
- [ ] 접근성 요구사항을 검토했는가?
- [ ] 브라우저/디바이스 호환성이 정의되었는가?

### 7. 기술 스펙
- [ ] 기술 스택이 명확히 정의되었는가?
- [ ] 아키텍처 다이어그램이 포함되었는가?
- [ ] 외부 서비스 의존성이 나열되었는가?
- [ ] 기술 선택의 근거가 문서화되었는가?

### 8. UI/UX 설계
- [ ] 사용자 플로우가 시각화되었는가?
- [ ] 주요 화면의 와이어프레임이 있는가?
- [ ] 디자인 시스템/가이드라인을 따르는가?
- [ ] 모바일/반응형 디자인을 고려했는가?

### 9. 데이터 모델
- [ ] 모든 주요 엔티티가 정의되었는가?
- [ ] 필드 타입과 제약사항이 명시되었는가?
- [ ] 엔티티 간 관계가 명확한가?
- [ ] ERD 다이어그램이 포함되었는가?
- [ ] 데이터 마이그레이션 계획이 있는가?

### 10. API 명세
- [ ] 모든 엔드포인트가 문서화되었는가?
- [ ] Request/Response 예시가 포함되었는가?
- [ ] 에러 케이스와 응답 코드가 정의되었는가?
- [ ] API 버전 관리 전략이 있는가?
- [ ] 인증/인가 방법이 명시되었는가?

### 11. 보안 및 권한
- [ ] 인증/인가 메커니즘이 정의되었는가?
- [ ] 역할별 권한이 명확히 구분되었는가?
- [ ] 개인정보 보호 방안이 있는가?
- [ ] 보안 취약점을 검토했는가?
- [ ] 컴플라이언스 요구사항을 확인했는가?

### 12. 성공 지표 (KPI)
- [ ] 비즈니스 메트릭이 측정 가능한가?
- [ ] 기술 메트릭이 정의되었는가?
- [ ] 측정 방법과 도구가 명시되었는가?
- [ ] 목표값이 현실적이고 달성 가능한가?
- [ ] 모니터링 대시보드 계획이 있는가?

### 13. 일정 및 마일스톤
- [ ] 주요 마일스톤이 명확히 정의되었는가?
- [ ] 각 마일스톤의 산출물이 구체적인가?
- [ ] 목표 일정이 현실적인가?
- [ ] 버퍼 시간을 고려했는가?
- [ ] 의존성에 따른 일정 조정을 했는가?

### 14. 리스크 및 완화 방안
- [ ] 주요 리스크를 모두 파악했는가?
- [ ] 각 리스크의 영향도와 확률을 평가했는가?
- [ ] 모든 리스크에 대한 완화 방안이 있는가?
- [ ] 리스크별 담당자가 지정되었는가?
- [ ] 정기적인 리스크 리뷰 계획이 있는가?

### 15. 의존성
- [ ] 내부 시스템/팀 의존성을 파악했는가?
- [ ] 외부 서비스 의존성을 파악했는가?
- [ ] 의존성으로 인한 블로커를 확인했는가?
- [ ] 의존성 해결을 위한 계획이 있는가?

### 16. 참고 자료
- [ ] 관련 문서 링크를 포함했는가?
- [ ] 경쟁사 분석 자료를 참조했는가?
- [ ] 기술 문서나 라이브러리 레퍼런스를 포함했는가?

## 🔍 검토 단계 (Review)

### 자체 검토
- [ ] 맞춤법과 문법을 확인했는가?
- [ ] 일관된 용어와 표현을 사용했는가?
- [ ] 모든 섹션이 완성되었는가?
- [ ] 이미지와 다이어그램이 올바르게 표시되는가?
- [ ] 링크가 정상적으로 작동하는가?

### 명확성 검토
- [ ] 기술적 배경이 없는 사람도 이해할 수 있는가?
- [ ] 모호한 표현이나 가정이 없는가?
- [ ] 전문 용어에 대한 설명이 있는가?
- [ ] 요구사항이 구체적이고 실행 가능한가?

### 완성도 검토
- [ ] 모든 필수 섹션이 작성되었는가?
- [ ] 요구사항이 SMART(구체적, 측정가능, 달성가능, 관련성, 시간제한)한가?
- [ ] 엣지 케이스와 예외 상황을 고려했는가?
- [ ] 테스트 계획이 포함되었는가?

### 이해관계자 검토
- [ ] 프로덕트 매니저의 검토를 받았는가?
- [ ] 기술 리드의 검토를 받았는가?
- [ ] 디자이너의 검토를 받았는가?
- [ ] 주요 이해관계자의 피드백을 반영했는가?
- [ ] 모든 검토 의견을 문서화했는가?

## ✅ 승인 전 (Pre-Approval)

### 최종 확인
- [ ] 문서 상태를 "🔵 Review"로 변경했는가?
- [ ] 모든 피드백이 반영되었는가?
- [ ] 변경 이력이 업데이트되었는가?
- [ ] 버전 번호가 올바른가?
- [ ] 승인자 목록이 정확한가?

### 배포 준비
- [ ] 관련 팀에게 PRD를 공유했는가?
- [ ] 킥오프 미팅 일정을 잡았는가?
- [ ] 프로젝트 트래킹 시스템(Jira, Linear 등)을 설정했는가?
- [ ] 초기 이슈/티켓을 생성했는가?

---

## 🚀 승인 후 (Post-Approval)

### 문서 관리
- [ ] 문서 상태를 "🟢 Approved"로 변경했는가?
- [ ] 승인 날짜와 서명을 기록했는가?
- [ ] 최종 버전을 적절한 위치에 저장했는가?
- [ ] 관련 팀에게 승인 사실을 알렸는가?

### 프로젝트 시작
- [ ] 킥오프 미팅을 진행했는가?
- [ ] 개발팀에게 PRD를 상세히 설명했는가?
- [ ] 초기 스프린트 계획을 수립했는가?
- [ ] 진행 상황 추적 프로세스를 설정했는가?

### 지속적 관리
- [ ] 정기적인 PRD 업데이트 일정을 설정했는가?
- [ ] 변경사항 발생 시 업데이트 프로세스가 있는가?
- [ ] 프로젝트 진행에 따라 문서 상태를 업데이트하는가?

## 💡 팁

### 효과적인 PRD 작성을 위한 팁
1. **간결함**: 불필요한 내용은 제거하고 핵심에 집중
2. **시각화**: 복잡한 내용은 다이어그램으로 표현
3. **구체성**: "빠르게", "많이" 같은 모호한 표현 대신 구체적인 수치 사용
4. **일관성**: 용어와 스타일을 문서 전체에서 일관되게 유지
5. **검증 가능성**: 모든 요구사항이 테스트 가능하도록 작성

### 흔한 실수 피하기
- ❌ 너무 기술적이거나 너무 추상적인 내용
- ❌ 우선순위 없이 모든 기능을 동등하게 다루기
- ❌ 측정 불가능한 목표 설정
- ❌ 이해관계자 피드백을 받지 않고 진행
- ❌ 엣지 케이스와 에러 처리 누락
