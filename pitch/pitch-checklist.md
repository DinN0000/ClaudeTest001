# Pitch 작성 체크리스트

Pitch를 작성하거나 검토할 때 다음 체크리스트를 활용하세요.

---

## 📝 작성 전 (Pre-Writing)

### 준비 단계
- [ ] 프로젝트의 목적과 배경을 명확히 이해했는가?
- [ ] 핵심 이해관계자를 파악하고 연락했는가?
- [ ] 유사한 프로젝트나 경쟁사 사례를 조사했는가?
- [ ] 템플릿을 적절한 위치에 복사했는가? (feature/bugfix/improvement)

### 파일 설정
- [ ] 파일명이 네이밍 컨벤션을 따르는가? (`프로젝트명_MM-DD-YYYY_Pitch.md`)
- [ ] 문서 상단의 메타데이터(상태, 작성자, 날짜)를 작성했는가?
- [ ] 초기 상태를 "🟡 Draft"로 설정했는가?

---

## 📄 내용 작성 (Content)

### 1. Background 섹션

- [ ] 프로젝트가 필요한 배경을 명확히 설명했는가?
- [ ] 해결하려는 문제가 구체적으로 정의되었는가?
- [ ] 비즈니스적/기술적 임팩트가 설명되었는가?
- [ ] 내용이 간결한가? (3-5문장)
- [ ] 데이터나 수치가 있다면 포함했는가?

**좋은 예시:**
> "람다256이 보유한 블록체인 기술(MPC Wallet SDK, Paymaster 등)을 활용하여 미래에셋 증권의 자산 포트폴리오를 겸한 web2 친화적인 web3 지갑을 Pilot으로 개발하여 제공한다."

---

### 2. Solution 섹션

#### 제품 개요 - 핵심 가치
- [ ] 핵심 가치가 표 형태로 정리되었는가?
- [ ] 각 핵심 가치가 간단명료한가?
- [ ] 3-5개의 핵심 가치로 정리되었는가?

**좋은 예시:**
| 핵심 가치 | 설명 |
| --- | --- |
| 포트폴리오 통합 | Web2 자산과 Web3 자산을 한 화면에서 조회 |
| 보안 및 컴플라이언스 | MPC 기반 키/서명 관리, KYC/AML 지원 |

#### 차별화 기능
- [ ] 각 차별화 기능이 명확히 구분되어 있는가?
- [ ] **문제 인식**이 구체적으로 작성되었는가?
- [ ] **해결 방식**이 명확하게 설명되었는가?
- [ ] **효과**가 측정 가능하거나 구체적인가?
- [ ] 각 기능이 `<aside>` 블록으로 구분되어 있는가?

**좋은 예시:**
```markdown
<aside>

**MPC 보안지갑**

**문제 인식:** 일반적인 Web3 지갑은 사용자가 Private Key를 단독 보관하며,
분실 시 복구가 불가능합니다.

**해결 방식:** 키를 다중으로 분할(MPC)하여 브라우저/서버 등에 분산 보관합니다.

**효과:**
1. '키 분실 시 복구 불가' 문제를 해결한다.
2. 악성 컨트랙트 서명으로 인한 해킹 리스크를 사전 차단한다.

</aside>
```

### 3. User Story 섹션

#### Story 구성
- [ ] 각 Story가 기능 단위로 구성되었는가?
- [ ] Story 제목이 `[카테고리]` + 설명 형태인가?
- [ ] 카테고리가 명확한가? (예: 포트폴리오, 지갑, 설정 등)
- [ ] 사용자 관점에서 작성되었는가?

**좋은 예시:**
```markdown
### Story 1: `[포트폴리오]` 사용자는 '포트폴리오'에서 web2 + web3 종합 자산을 확인한다.
```

#### Test Case 작성
- [ ] 각 Story에 Test Case가 포함되어 있는가?
- [ ] Test Case가 `<aside>` 블록 안에 있는가?
- [ ] 각 Test Case가 UI 상호작용을 상상할 수 있는가?
- [ ] "사용자는 ~을 할 수 있다" 형태로 작성되었는가?
- [ ] 각 Test Case가 하나의 명확한 동작만 설명하는가?
- [ ] 버튼 클릭, 입력, 확인, 선택 등 구체적인 행동이 포함되었는가?

**좋은 예시:**
```markdown
<aside>

**Test Case**

1. 사용자는 ID/PW를 통해 로그인할 수 있다.
2. 사용자는 포트폴리오 홈에서 총자산(KRW 기준)을 확인할 수 있다.
3. 사용자는 포트폴리오 홈에서 증권 자산(더미)을 확인할 수 있다.
4. 사용자는 포트폴리오 홈에서 지갑 생성 또는 지갑 자산을 확인할 수 있다.

</aside>
```

**나쁜 예시:**
```markdown
- 로그인 기능이 작동한다. (모호함)
- 대시보드를 본다. (주체가 불명확)
- 데이터가 표시된다. (무엇을 어떻게?)
```

## 🔍 검토 단계 (Review)

### 전체 구조 검토
- [ ] Background, Solution, User Story 3개 섹션이 모두 작성되었는가?
- [ ] 각 섹션이 균형 있게 작성되었는가?
- [ ] 문서의 흐름이 자연스러운가?

### Background 검토
- [ ] 간결하고 명확한가? (3-5문장)
- [ ] 기술 배경이 없는 사람도 이해할 수 있는가?
- [ ] 왜 이 프로젝트를 해야 하는지 명확한가?

### Solution 검토
- [ ] 핵심 가치가 표 형태로 정리되었는가?
- [ ] 차별화 기능이 문제→해결→효과 구조를 따르는가?
- [ ] 각 차별화 기능이 독립적으로 이해 가능한가?
- [ ] 기술 용어에 대한 설명이 충분한가?

### User Story 검토
- [ ] 모든 주요 기능이 Story로 작성되었는가?
- [ ] Story가 개발 가능한 수준으로 구체적인가?
- [ ] Test Case가 QA 가능한 수준으로 명확한가?
- [ ] Test Case에서 UI 플로우를 상상할 수 있는가?

### 명확성 검토
- [ ] 맞춤법과 문법을 확인했는가?
- [ ] 일관된 용어를 사용했는가?
- [ ] 모호한 표현이 없는가?

## 💡 팁

### 효과적인 Pitch 작성 팁

**Background**
- 간결함이 생명: 3-5문장으로 핵심만
- 숫자와 데이터로 임팩트 강조
- 비즈니스 가치 중심으로 작성

**Solution - 제품 개요**
- 핵심 가치는 표 형태로 한눈에
- 3-5개로 제한 (너무 많으면 산만)
- 각 항목은 한 줄로 요약

**Solution - 차별화 기능**
- 문제→해결→효과 구조 엄수
- 각 효과는 측정 가능하게
- 기술 용어는 반드시 설명 추가

**User Story**
- 한 Story는 한 기능만
- 카테고리를 명확히
- 개발자가 바로 개발 가능한 수준

**Test Case**
- UI 상호작용 중심
- "사용자는 ~을 할 수 있다" 형태
- 각 케이스는 하나의 동작만
- QA가 바로 테스트 가능한 수준

### 흔한 실수 피하기

**Background**
- ❌ 너무 길고 장황함
- ❌ 기술적 디테일에 집중
- ✅ 간결하고 비즈니스 임팩트 중심

**Solution**
- ❌ 핵심 가치를 문장으로 길게 설명
- ❌ 문제만 나열하고 해결 방식 누락
- ✅ 표 형태 + 문제→해결→효과 구조

**User Story**
- ❌ "로그인 기능", "대시보드" 같은 모호한 표현
- ❌ 기술 구현 중심 작성
- ✅ 사용자 행동 중심, UI 상호작용 명확히

**Test Case**
- ❌ "기능이 작동한다"
- ❌ 여러 동작을 한 케이스에 혼합
- ✅ "사용자는 X 버튼을 클릭하여 Y를 할 수 있다"

## 📊 품질 기준

### 우수한 Pitch의 조건
1. Background를 읽으면 왜 하는지 즉시 이해됨
2. Solution을 읽으면 어떻게 해결하는지 명확함
3. User Story를 읽으면 개발자가 바로 개발 시작 가능
4. Test Case를 읽으면 QA가 바로 테스트 시작 가능
5. 전체 문서가 간결하고 명확함 (불필요한 내용 없음)

### 개선이 필요한 Pitch
1. Background가 너무 길거나 모호함
2. Solution에 문제만 있고 해결 방식이 불명확함
3. User Story가 추상적이거나 기술 구현 중심
4. Test Case가 모호하거나 UI 플로우 상상 불가
5. 전체적으로 산만하거나 중복이 많음
